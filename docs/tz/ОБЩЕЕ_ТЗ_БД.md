# ОБЩЕЕ ТЗ: Подключение БД (Supabase) — Все чаты

## Обзор

Фаза 2 разработки Android-приложения iLeader: замена mock-данных на реальные запросы к Supabase.

**Что делает координатор (главный чат) ПЕРЕД вашей работой:**
- Создаёт `SupabaseModule.kt` (singleton клиент)
- Создаёт базовые DTO (`ProfileDto`, `TournamentDto`, `LocationDto`, `TeamDto`, etc.)
- Обновляет `AuthViewModel.kt` на реальную авторизацию через Supabase Auth
- Создаёт `BuildConfig` поля для URL и ключа
- Создаёт базовый `UiState` sealed class

**Что делает каждый чат:**
- Создаёт Repository для своей роли
- Создаёт ViewModel(ы) для своих экранов
- Обновляет экраны: убирает прямые ссылки на MockData, подключает ViewModel

---

## Архитектура

```
Screen (Composable)
  ↓ observes StateFlow<UiState<T>>
ViewModel
  ↓ calls suspend fun
Repository
  ↓ calls Supabase SDK
SupabaseModule.client (singleton)
  ↓ HTTP via Ktor/OkHttp
Supabase PostgreSQL (с RLS)
```

### 4. `data/remote/dto/` — DTO-классы

DTO — это data class с `@Serializable` и `@SerialName` для маппинга на колонки Supabase.

**ПРАВИЛА DTO:**
- Имя = название таблицы + `Dto` (ProfileDto, TournamentDto)
- Все поля nullable (кроме `id`) — база может вернуть null
- `@SerialName("column_name")` для snake_case → camelCase
- Вспомогательный метод `toDomain()` для конвертации в модель приложения

## Supabase-kt: Синтаксис запросов

### Чтение (SELECT)

```kotlin
import io.github.jan.supabase.postgrest.from
import io.github.jan.supabase.postgrest.query.Columns

// Простой SELECT
val profiles = SupabaseModule.client.from("profiles")
    .select()
    .decodeList<ProfileDto>()

// С фильтрами
val athletes = SupabaseModule.client.from("profiles")
    .select {
        filter { eq("primary_role_id", athleteRoleId) }
    }
    .decodeList<ProfileDto>()

// С JOIN (select конкретных колонок + вложенные таблицы)
val tournaments = SupabaseModule.client.from("tournaments")
    .select(Columns.raw("*, sports(name), locations(name, city), profiles!organizer_id(name)"))
    .decodeList<TournamentWithDetailsDto>()

// Один элемент
val profile = SupabaseModule.client.from("profiles")
    .select { filter { eq("id", userId) } }
    .decodeSingle<ProfileDto>()

// Из View
val stats = SupabaseModule.client.from("v_organizer_stats")
    .select { filter { eq("organizer_id", userId) } }
    .decodeSingle<OrganizerStatsDto>()

// С сортировкой и лимитом
val recent = SupabaseModule.client.from("tournaments")
    .select {
        filter { eq("status", "registration_open") }
        order("start_date", Order.ASCENDING)
        limit(10)
    }
    .decodeList<TournamentDto>()

// RPC (вызов функции)
val hasRole = SupabaseModule.client.postgrest.rpc("has_role", mapOf("check_role" to "athlete"))
    .decodeSingle<Boolean>()
```

## Паттерн ViewModel + Repository

### Repository (пример)

```kotlin
package com.ileader.app.data.repository

import com.ileader.app.data.remote.SupabaseModule
import com.ileader.app.data.remote.dto.ProfileDto
import io.github.jan.supabase.postgrest.from

class AthleteRepository {
    private val client = SupabaseModule.client

    suspend fun getProfile(userId: String): ProfileDto {
        return client.from("profiles")
            .select { filter { eq("id", userId) } }
            .decodeSingle<ProfileDto>()
    }

    suspend fun getUpcomingTournaments(userId: String): List<TournamentDto> {
        return client.from("tournament_participants")
            .select(Columns.raw("tournaments(*, sports(name), locations(name, city))"))
            {
                filter {
                    eq("athlete_id", userId)
                    eq("tournaments.status", "registration_open")
                }
            }
            .decodeList<TournamentDto>()
    }
}
```

### ViewModel (пример)

```kotlin
package com.ileader.app.ui.viewmodels

import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import com.ileader.app.data.remote.UiState
import com.ileader.app.data.repository.AthleteRepository
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.launch

class AthleteDashboardViewModel : ViewModel() {
    private val repo = AthleteRepository()

    private val _state = MutableStateFlow<UiState<AthleteDashboardData>>(UiState.Loading)
    val state: StateFlow<UiState<AthleteDashboardData>> = _state

    fun load(userId: String) {
        viewModelScope.launch {
            _state.value = UiState.Loading
            try {
                val profile = repo.getProfile(userId)
                val tournaments = repo.getUpcomingTournaments(userId)
                val stats = repo.getStats(userId)
                _state.value = UiState.Success(
                    AthleteDashboardData(profile, tournaments, stats)
                )
            } catch (e: Exception) {
                _state.value = UiState.Error(e.message ?: "Ошибка загрузки")
            }
        }
    }
}
```

### Использование в Screen

```kotlin
@Composable
fun AthleteDashboardScreen(user: User) {
    val viewModel: AthleteDashboardViewModel = viewModel()
    val state by viewModel.state.collectAsState()

    LaunchedEffect(user.id) { viewModel.load(user.id) }

    when (val s = state) {
        is UiState.Loading -> LoadingScreen()
        is UiState.Error -> ErrorScreen(s.message) { viewModel.load(user.id) }
        is UiState.Success -> {
            // Текущая верстка, но данные из s.data вместо MockData
        }
    }
}
```

## Правила для всех чатов

1. **НЕ удаляйте mock-данные!** Оставьте файлы `*MockData.kt` — они нужны как fallback и для тестов. Создавайте Repository рядом.

2. **НЕ меняйте UI-верстку** — меняется только источник данных (MockData → ViewModel).

3. **Все запросы в try/catch** — Supabase может выбросить исключение (сеть, RLS, etc.).

4. **userId получаем из `User.id`** который приходит в параметрах Screen.

5. **Все DTO — `@Serializable`** с `@SerialName` для snake_case полей.

6. **Используйте `viewModel()` из `lifecycle-viewmodel-compose`** для получения ViewModel в Composable.

7. **Один Repository на роль** (AthleteRepository, TrainerRepository, etc.). Если нужно — можно разбить на несколько (TrainerTeamRepository, TrainerTournamentRepository).

8. **Формат дат** — Supabase возвращает ISO 8601 (`2026-03-15T10:00:00+00:00`). Парсите через `java.time.Instant` или `java.time.LocalDate`.

9. **Не используйте Realtime на первом этапе** — только REST запросы. Realtime добавим позже.

10. **Каждый чат создаёт свой файл(ы):**
    - `data/repository/{Role}Repository.kt`
    - `ui/viewmodels/{Role}{Screen}ViewModel.kt` (по одному на экран или группу)
    - Обновляет свои Screen-файлы
